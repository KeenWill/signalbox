# Invariant catalog

This catalog is the normative law catalog: each INV row is the statement of
record, and the INV-tagged tests cited in its enforcement column are the
enforcement of record. Implemented mechanics behind each law are described by
the living spec under [docs/spec](spec/README.md), and each row links the spec
page(s) that own its topic. "Accepted" records current direction; "Provisional"
means the constraint or its exact boundary needs a decision. Each invariant has
a primary classification; actual enforcement may use several layers, and the
enforcement column accumulates citations of the tests, types, and constraints
that enforce the invariant as slices are implemented.

Cross-cutting evidence-bearing reconstitution is owned by
[turn-lifecycle-and-scheduling](spec/turn-lifecycle-and-scheduling.md) and
[persistence-protocol](spec/persistence-protocol.md).

## Review rule

A feature proposal must cite the invariants it relies on and add a scenario when
it introduces a new lifecycle edge. If an implementation cannot enforce an
accepted invariant, the change must revise the design — updating this catalog
and the owning [docs/spec](spec/README.md) page — rather than weakening the
invariant implicitly.
