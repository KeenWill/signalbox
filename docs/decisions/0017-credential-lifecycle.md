# ADR-0017: Provider and integration credential lifecycle

- Date: 2026-07-17
- Supersedes: none
- Superseded by: none
- Depends on: the accepted foundation set ([ADR-0001](0001-domain-terminology-and-identity.md), [ADR-0003](0003-session-creation-and-transcript-ancestry.md), [ADR-0004](0004-turn-and-attempt-lifecycle.md), [ADR-0005](0005-model-call-retry-semantics.md), [ADR-0027](0027-input-delivery-lifecycle.md)); consistent with accepted [ADR-0022](0022-persistence-representation.md), whose schema map contains no credential table and which explicitly did not decide credential storage
- Refines: the enforcement boundary of INV-035 (see the INV-035 enforcement index)
- Resolves: the ["Provider and integration credential storage and rotation"](../open-questions.md#identity-credentials-and-resource-governance-reserved-adr-0015-through-adr-0018) question (reserved ADR-0017)
- Decision questions: channel ownership between 1Password and sops-age-in-git so no secret has two sources of truth; the delivery path into the hub process with rotation behavior; the in-process access boundary that enforces INV-035; failure behavior when a credential or its manager is unavailable at startup versus during operation

## Context

The accepted records already fix every semantic constraint this decision must satisfy without choosing a mechanism. INV-035 states that "Provider credentials remain under hub control in the initial architecture and are not sent to clients or general-purpose runners," its enforcement column reserving "Secret access boundary and adapter interfaces; sanitized logs and protocol schemas." The [architecture's](../architecture.md) sources-of-truth table records provider credentials as "Hub-controlled secret storage; exact mechanism open" and "Never client or runner session state." ADR-0027 places provider credentials outside `EffectiveConfiguration` as "late-bound operational facts" that "must be recorded when relevant but may not change the requested model selection or authorize retry/fallback." The open question's recorded leaning is "Secret values stay outside ordinary session records, exposed only to the responsible hub adapter." Scenario S15 executes hub-local integrations "under central credentials," and S02, S20–S23 all require provider calls whose provenance is durable while their authentication material is not.

The deployment context is now concrete. The hub deploys to a Kubernetes cluster the owner controls. The owner operates 1Password with the 1Password Kubernetes operator, which syncs vault items into native Kubernetes Secrets, and separately uses sops with age encryption for configuration committed to git. This record decides how those two existing channels divide ownership of Signalbox's secrets and how a credential travels from its source of truth into a provider or integration adapter. It selects no new dependency: the operator and sops run outside the Signalbox binary, and the delivery path below is consumable with the standard library.

One candidate from the original option list is rejected up front: **encrypted database records** — provider or integration credentials stored in Postgres in any form, plaintext, hashed, or encrypted. First, Postgres is the canonical durable store whose backups must be ordinary and restorable; credential bytes in rows flow into every base backup, WAL segment, and logical replica, so one leaked backup becomes a credential incident and every rotation acquires a backup-retention tail. Second, key management is circular: encrypted rows need a decryption key the hub must obtain from somewhere else anyway, so the database adds a second secret store without removing the first delivery problem. Third, INV-035's boundary is weakest exactly there: values traveling through SQL, driver logs, and query text are the opposite of a narrow "secret access boundary," and ADR-0022's record/domain mapping would need credential-aware exceptions. The rest of this record therefore assumes Postgres holds non-secret credential references at most, never values.

## Decision

### Two channels, one source of truth per secret

The secret classes this record governs — provider and integration runtime credentials, plus the bootstrap and deployment material the channels themselves depend on — are each assigned to exactly one channel, determined by class. Secret classes reserved to other records, notably owner client authentication (ADR-0015) and runner enrollment (ADR-0016), are outside this rule and stay undecided here; those records may reuse the classification without being bound by it. The rule is:

- **1Password owns runtime credentials**: every runtime credential the running hub consumes whose value can change without a deployment. This includes provider credentials (model-provider API keys) and integration credentials (tokens used by hub-local tool adapters, S15). The vault-item material a credential reference resolves to is its source of truth; rotation is an edit to that material. The mapping from reference to vault items — whether one item per credential or shared items carrying several fields — is deployment layout this record leaves open (see non-decisions).
- **sops-age-in-git owns deployment and bootstrap configuration**: every secret that must exist before the 1Password channel can operate, or that configures the deployment itself. This includes the operator's own credential — the 1Password Connect credentials bundle and Connect token, or a service-account token where the operator supports one — plus any other secret material the cluster needs before the operator runs. The encrypted file in git is the source of truth; rotation is an edit, re-encryption, commit, and redeploy, so bootstrap rotation history is git history. Changing the age recipient set is a distinct operation (sops `updatekeys`, or `rotate` when a recipient is removed) that re-encrypts file keys without changing any secret value.

The two clauses are exclusive and exhaustive for the secret classes this record governs — provider and integration runtime credentials plus the bootstrap and deployment material the channels depend on — not for every cluster-delivered secret; secret classes reserved to other records (owner client authentication, ADR-0015; runner enrollment, ADR-0016) stay outside this split even when a future deployment delivers them through Kubernetes. Maintaining the same value in both channels is a defect, not redundancy: a secret has exactly one place where editing it is rotation. Kubernetes Secret objects are **delivery artifacts** of whichever channel produced them, never sources of truth; an operator-synced Secret restates a vault item, and a sops-decrypted Secret restates a git file. Hand-editing a delivery artifact is likewise a defect because the next sync or deploy overwrites it.

The bootstrap chain is acyclic: the owner-held age identity decrypts the sops channel; the sops channel delivers the operator's own credential; the operator syncs the 1Password channel; the hub consumes its delivery artifacts. The age identity is root material under owner custody outside git and outside operator sync. Where the owner's deploy process stores or supplies it (manual decrypt-and-apply, CI, or a GitOps controller holding a decryption Secret delivered once by hand) is deployment mechanics this record does not fix, provided no cluster workload can reach the age identity through the 1Password channel — that dependency would close the loop the split exists to prevent.

### Delivery path into the hub process

Runtime credentials reach the hub as an **operator-synced native Kubernetes Secret mounted as a volume**, consumed by file read. The three candidate paths differ exactly in rotation behavior:

| Path | Value fixed at | Rotation behavior |
| --- | --- | --- |
| Operator-synced Secret as environment variables | Container start | A changed value requires a pod restart (the operator's auto-restart annotation performs a rolling restart on sync). Every rotation abandons the process: prior-process nonterminal turn attempts end `Lost` under the INV-034 startup scan, so rotating an unrelated provider's key terminalizes in-flight work |
| Operator-synced Secret as a mounted volume (selected) | Each read | The kubelet refreshes mounted Secret contents on its sync period (about a minute by default; `subPath` mounts never update and are prohibited here). The hub reads the file per use, so rotation propagates without restart and without touching in-flight work |
| Direct 1Password Connect or service-account API access | Each fetch | Freshest possible value, but every read is a live network dependency: a Connect or 1Password outage fails new provider calls even though a last-known-good value exists, service accounts carry documented hourly and daily request quotas, and the hub still needs a bootstrap token delivered by one of the other two paths |

The selected path bounds rotation propagation by the operator's polling interval (600 seconds by default, configurable) plus the kubelet sync period — minutes with defaults, tunable where that matters. The hub's deployment must explicitly disable auto-restart rather than merely omit the annotation: the 1Password operator inherits auto-restart from operator- or namespace-level scopes unless a downstream scope overrides it ([docs](https://www.1password.dev/k8s/operator#enable-automatic-redeployment)), so the hub Deployment (or its `OnePasswordItem`) must set `operator.1password.io/auto-restart: "false"`, or otherwise guarantee no inherited scope restarts the hub. Restart-per-rotation is exactly the behavior the mounted path avoids, and honest classification of deliberately abandoned work should not be routine rotation cost.

Rotation follows an overlap discipline the provider side must honor: install the new value in the vault item, wait out the propagation bound plus the longest expected in-flight provider call, then revoke the old value at the provider. A call authenticated before rotation completes on the old value; a call prepared after propagation reads the new one. Where a provider supports only one active key at a time, rotation has an honest failure window in which sends fail as known failures — narrower propagation configuration, not silent retry, is the mitigation.

### In-process access boundary

A **credential reference** is a non-secret durable name for exactly one logically distinct credential (for example, one provider account's API key). Rotating the value preserves the reference; a credential with different scope or ownership gets a new reference. References are safe in configuration, logs, records, and error text; values are safe nowhere but the adapter boundary below.

Non-secret deployment configuration maps each configured provider selection to one credential reference by its **provider component** — the provider-account dimension of the selection, not the full provider/model pair — and each hub-local integration adapter to its references. The exact target ADR-0005 pins per turn (the deployment side of ADR-0027's `DirectModelSelection`, carried today as the provider/model identity `ResolvedProviderTarget`/`ProviderModelIdentity` in [`crates/domain/src/model_call.rs`](../../crates/domain/src/model_call.rs)) names a full provider/model identity, so the reference is keyed by that target's provider component: one provider account's key serves every model under that account, and no distinct per-model credential is implied. How the provider component is projected from the normalized identity is the provider-provenance decision's (reserved ADR-0007), which owns provider-identity normalization. The credential reference for a pinned target is captured as a durable fact when the attempt pins its target, exactly as the target itself freezes under ADR-0005 — the exact storage and provenance shape of that snapshot is deferred to the persistence representation (ADR-0022), but some durable reference must be recorded at the pinning boundary so it survives a restart. Send preparation and post-restart recovery resolve the value for that already-fixed, durably recorded reference and never re-derive it from the mutable reference-to-provider mapping, so a call authenticates under the reference the accepted work pinned, never under a mapping that changed afterward. Remapping a provider component to a different credential reference is therefore a configuration change that governs only turns pinned after it takes effect — a target/configuration change, not rotation — while rotation remains a new value under the same fixed reference. The hub exposes one **credential access port**: given a credential reference, it reads the mounted delivery file and returns the value as an opaque in-process boundary value scoped to one physical request. Precisely:

1. A provider adapter obtains a credential only through the port, only by reference, and only during send preparation of a specific model call (between the durable `Prepared` record and the send authorization that moves it `InFlight` under ADR-0005); integration adapters use the same port at the equivalent effect boundary of a tool attempt.
2. The boundary value type exposes credential bytes only to the code that authenticates the outbound request. It offers no general-purpose formatting, serialization, or persistence path; this is a boundary-only type in the sense of INV-002, and `crates/domain` defines no credential-bearing type (none exists today, and none may be added).
3. No credential value enters a domain type, a storage record or column, a protocol message, a context frontier or semantic history entry, a client snapshot, a log line, or error text. Port and adapter errors carry the credential reference and a typed failure class, never the value.
4. Durable records carry at most a credential reference, never a value. The reference a pinned target resolves to must be durably recorded at the pinning boundary so it survives a restart; the exact record and column that carry it — for instance whether the model-call record itself names it — is the persistence representation's (ADR-0022), and how the reference is derived from the provider target is the provider-provenance decision's (reserved ADR-0007).

Clause phrasing intended for INV-035's enforcement column when the provider adapter slice lands: *credential values reach the hub only through the credential access port keyed by non-secret credential references; the boundary value type has no formatting, serialization, or persistence path; storage and wire mappings accept references only; adapter errors and logs carry reference and typed failure class — INV-035-tagged boundary and log-sanitization tests.*

### Failure behavior

**At startup.** The INV-034 startup scan classifies durable evidence and "issues no semantic effects," so startup recovery never reads a credential; recovery of acknowledged work cannot be blocked by any provider's configuration. To keep that true the credential mount must not gate hub boot: the hub must start and run its startup scan even when the operator-synced Secret object is absent or individual reference files are missing — during a restore, a deleted Secret, or a first sync that has not yet completed. The delivery volume is therefore configured so its absence does not block scheduling (an `optional` Secret volume, or an equivalent that lets the pod start without the Secret present), and the hub performs no startup preflight over individual credentials and does not refuse to start because a reference is missing from the mounted files. An absent or never-bootstrapped credential channel stays a visible operational condition, but it is surfaced at the effect boundary that needs the credential, not by blocking startup recovery of acknowledged work.

**During operation.** A failed read at send preparation — reference unmapped, file or key absent, value unreadable — means preparation cannot proceed: the call takes ADR-0005's existing `Prepared` to `Terminal(KnownFailed)` edge, the attempt ends `WithoutStop(KnownFailure)`, and the turn fails, with no automatic retry and no fallback, exactly as any other send-preparation failure. Credential presence is not consulted by target resolution and is not part of ADR-0027's static pre-activation validation of a frozen configuration; both operate on frozen semantic meaning, and admitting mutable credential state there would smuggle a late-bound operational fact into boundaries ADR-0027 deliberately excludes it from. A provider rejecting the credential after send (an authentication error mid-request or mid-stream) is ordinary provider outcome evidence classified under the open provider-contract thresholds of ADR-0004 and ADR-0005; this record adds no classification rule.

**Manager unavailability.** A 1Password or Connect outage pauses sync; the operator-synced Secret is a durable cluster object holding the last-synced values, so the running hub and hub restarts continue on those values and only rotation propagation is delayed. (The operator's retain-on-outage behavior is observed rather than documented and must be verified at deployment; see open questions.) The failure to avoid is ordering: revoking an old value at the provider while sync is down turns every affected send into a known failure, which is why the overlap discipline revokes last. A sops channel failure is a failed deployment, never a running-hub event.

## Terminology

- **Credential:** A secret value that authenticates the hub to an external provider or integration.
- **Credential reference:** The non-secret durable name for exactly one logically distinct credential; stable across rotation.
- **Channel:** A source of truth plus its delivery mechanism into the cluster.
- **Delivery artifact:** A Kubernetes Secret or mounted file restating a channel's source of truth; never authoritative.
- **Rotation:** Replacing a credential's value under the same reference at its source of truth.

## Invariants

- INV-035: this record defines the enforcement boundary (access port, reference-only records, sanitized errors); the catalog's enforcement index links here without restating these rules, and the test-backed enforcement wording lands with the provider adapter slice.
- INV-002 / INV-005: the credential boundary value is a purpose-specific adapter representation; it appears in no domain, storage, wire, or presentation type.
- INV-008 and ADR-0027's configuration boundary: credential state never enters `EffectiveConfiguration`, provenance, or acceptance-time derivation; rotation changes no frozen configuration and no turn identity.
- INV-014 / ADR-0005: credential failures map onto existing call, attempt, and turn edges; no new call state, disposition, or retry is introduced.
- INV-017 / INV-018: a credential failure authorizes no fallback or substitution.

## Strongest alternative

Direct 1Password access from the hub at startup or per call, via Connect or a service account, skipping the operator and native Secrets. It is the freshest path and removes the synced Secret as an intermediate artifact.

It is rejected as the baseline because it converts every credential read into a live external dependency with documented service-account rate quotas, makes provider calls fail during a manager outage that the mounted path survives on last-synced values, requires a 1Password client dependency in the hub, and still needs its own bootstrap token delivered by another channel — so it adds a failure mode without removing the bootstrap problem. The access port deliberately isolates this choice: a later decision could adopt direct access behind the same port without touching domain semantics.

## Rejected alternatives

- **Encrypted database records.** Rejected up front in Context: backup and replication surface, key-management circularity, and the weakest possible INV-035 boundary.
- **Environment-variable delivery.** Values fix at container start, so every rotation forces a restart that ends in-flight attempts `Lost`; rotation of one provider's key should not be a lifecycle event for unrelated work.
- **sops-age as the runtime-credential channel.** Rotation would require commit-and-redeploy (a restart, as above), and running both channels for one class invites two sources of truth for the same value.
- **Hand-managed Kubernetes Secrets as sources of truth.** No audited home for edits, no rotation trail, and drift between operator-managed and hand-managed artifacts.
- **Credential values in `EffectiveConfiguration`, provenance, or session records.** Contradicts ADR-0027's explicit exclusion and the recorded leaning; would freeze secrets into immutable durable state that INV-008 forbids rewriting.
- **`subPath` mounts or the auto-restart annotation for the hub.** `subPath` silently never updates; auto-restart — whether set on the hub or inherited from an operator- or namespace-level scope the hub does not explicitly override — reintroduces restart-per-rotation.
- **A required credential volume that gates hub boot.** Making the mounted Secret required so the kubelet refuses to start the container without it turns a missing or unsynced credential into a boot failure, which blocks the INV-034 startup scan and so blocks recovery of acknowledged work on unrelated providers — the opposite of startup recovery being credential-independent.

## Consequences

Rotation of runtime credentials is an edit in 1Password with propagation bounded by two configurable intervals and no hub restart; bootstrap rotation is a reviewed git change. The hub gains a file read per provider send — negligible against a model call. Deployment gains real obligations: maintain the reference-to-item mapping and reference-to-provider-component mapping, scope the synced Secret's RBAC to the hub, explicitly disable auto-restart on the hub rather than rely on omission, mount the credential Secret optionally so its absence does not block hub boot, verify operator retain-on-outage behavior, and honor the revoke-last overlap discipline. Providers that cannot hold two active keys make rotation briefly and honestly lossy. No Rust dependency is added by this record; in-memory hygiene mechanics (zeroization or an equivalent) are an implementation choice whose crate selection follows the repository dependency rule.

## Scenario walkthroughs

- **S02 (stream a provider response):** Acceptance freezes configuration; eligibility fixes the frontier; the attempt pins the exact target; the call is created `Prepared`. Send preparation asks the port for the reference mapped from the pinned target's provider component, reads the mounted file, authenticates the request, and the call goes `InFlight`. Durable rows hold call, target, frontier — at most a reference, never a value. If the read fails, the call ends `KnownFailed`, the attempt `WithoutStop(KnownFailure)`, the turn `Failed`, and the visible error names the reference and failure class only.
- **S15 (hub-local tool):** The integration adapter obtains its credential through the same port at the tool attempt's effect boundary. Tool request, policy decision, approval, and result records carry no value; adapter loss keeps its known/ambiguous classification.
- **S20 (curated alias):** Alias meaning freezes at acceptance; resolution validates frozen meaning and pins the target without consulting credential state. A rotation between acceptance and send changes nothing semantic — the next send preparation simply reads the current value.
- **S21 (exact pinned model):** An unresolvable selection still fails before any call exists; a credential-read failure still fails after the call exists, as `KnownFailed`. The two failures stay distinct and separately visible, and neither silently chooses another model.
- **S22 (availability fallback):** A credential failure is a known failure and version one stops explicitly. This record adds credential failure to no future fallback allowlist — a missing or rejected credential is deployment misconfiguration, not the classified availability failure the recorded leaning contemplates; that decision stays with reserved ADR-0006.
- **S23 (safety refusal):** A refusal presupposes an authenticated, completed exchange; credential state plays no role in refusal classification. A rotation-window rejection mid-stream is provider outcome evidence under the ordinary rules, not a refusal.
- **Hub restart (S03, S04):** The pod restarts and mounts the durable Secret — even during a 1Password outage, last-synced values are present. The startup scan reads no credential while it ends prior-process attempts `Lost` and classifies interrupted operations. New eligible work reads current values at its next send preparation. Under the rejected environment-variable path, this entire walkthrough would run on every rotation.

## Extension implications

Runner enrollment credentials (reserved ADR-0016) and owner client authentication (reserved ADR-0015) may reuse the channel-classification rule and the reference/value split, but their identity and revocation semantics are their own decisions. A future fallback decision (reserved ADR-0006) must not treat credential failure as availability without explicit argument. Future runners that need scoped credentials for runner-local tools would be a deliberate INV-035 revision through an ADR, not a delivery-path extension. Multi-owner deployment would add an owner namespace to references before anything else.

## Open questions

- Concrete reference naming and the representation of the reference-to-item and reference-to-provider-component mappings; encodings for selection keys remain with the provider-provenance decision (reserved ADR-0007).
- Whether the durable model-call record names the credential reference used (reserved ADR-0007 provenance scope).
- Integration-credential scoping per adapter and whether centrally hosted MCP is one adapter type (S15's remaining questions; tool ADRs).
- Per-credential usage limits and quota behavior (reserved ADR-0018).
- Provider-contract evidence thresholds for authentication rejections, within the open classification question of ADR-0004 and ADR-0005.
- Verification at deployment that the operator retains last-synced Secrets across manager outages, and the tuned values for polling and kubelet sync intervals.
- In-memory hygiene mechanics and any crate selection for them, at implementation time.

## Explicit non-decisions

This record does not decide at-rest protection of conversation content or database backups — that is persistence and retention scope (ADR-0022 and reserved ADR-0028/0029), named here so it is not lost, and nothing above weakens the need to decide it there. It does not decide runner enrollment, authentication, or revocation (reserved ADR-0016), or owner client authentication (reserved ADR-0015). It selects no Rust dependency, provider SDK, 1Password client library, GitOps tool, or zeroization crate; it writes no Kubernetes manifests and fixes no vault, item, or namespace layout beyond the channel-ownership rule; and it defines no provider error taxonomy. Mechanism names above (operator annotations, polling and sync intervals, sops commands) describe the owner's existing operational stack for falsifiability, not an API Signalbox owns.
